# DESIGN Review

対象: `/docs/DESIGN.md`
観点: 1) アーキテクチャ 2) エッジケース/エラーハンドリング 3) コスト/性能 4) セキュリティ 5) 代替案

## Findings (重大度順)

### 1. [High] Secrets の露出範囲が広い
- 問題: `env` を workflow/job 全体に置いており、`pip install` など不要ステップにも Secret が注入される。
- リスク: 依存パッケージやログ経由の漏えい面積が増える。
- 改善:
  - Secret は `python src/main.py` 実行 step の `env` に限定
  - `permissions: contents: read` を明示
  - `actions/*@vX` ではなく commit SHA pinning を採用

### 2. [High] 依存関係と Action のサプライチェーン対策が弱い
- 問題: `>=` 指定のみで再現性が低く、将来の破壊的変更や悪性アップデート耐性が弱い。
- 改善:
  - `requirements.txt` を厳密 pin（可能なら hash 付き）
  - Action は SHA pinning
  - Dependabot + CI テストで更新を管理

### 3. [High] 部分失敗時の品質劣化が無言で進む設計
- 問題: `skip ticker` や `use available metrics only` が多く、結果の信頼性低下を通知側で可視化しない。
- 改善:
  - 「成功/失敗銘柄数」「欠損指標数」「データ鮮度」を Slack に明示
  - 閾値以下のデータ品質なら「分析失敗」として送信を抑止

### 4. [Medium] LLM 出力契約が脆い（JSON修復1回のみ）
- 問題: プロンプト依存の JSON 生成は壊れやすい。修復1回では不十分。
- 改善:
  - JSON Schema バリデーション必須化
  - 失敗時は deterministic なテンプレート出力にフォールバック
  - 可能なら structured output API を利用

### 5. [Medium] Slack Block Kit 制限超過の考慮不足
- 問題: 保有銘柄数増加時に block 数・文字数制限に到達しうる。
- 改善:
  - 上限を見て自動分割送信
  - 詳細は gist/artifact へ、Slack は要約のみ

### 6. [Medium] コスト見積もりが楽観的
- 問題: Top 50 のフル指標を毎回送る設計は token 量が大きく、モデル更新で単価も変動する。
- 改善:
  - Claude入力は上位 N 件の圧縮特徴量に限定
  - monthly token 実測ログを取り、予算上限を超えたら縮退運転

### 7. [Medium] 市場カレンダー考慮が不足
- 問題: 日本の祝日・大納会短縮など非取引日判定がない。
- 改善:
  - JP 営業日カレンダーで実行可否を判定
  - 非取引日は「休場」通知のみにして API 呼び出しを抑制

### 8. [Low] Nikkei225 ハードコードは運用負債化しやすい
- 問題: 入替時にメンテ漏れすると分析母集団が陳腐化。
- 改善:
  - 半期自動更新ジョブ + 差分検知アラート
  - 取得元失敗時のみ静的リストへフォールバック

## 観点別コメント

### 1) アーキテクチャ
- 全体フローはシンプルで妥当。
- ただし「データ品質ゲート」「出力契約（schema）」「運用監視（メトリクス）」の3点が不足。

### 2) エッジケース/エラーハンドリング
- 休場日、データ欠損、部分失敗、Slack制限、LLM JSON破損への対策を強化すべき。
- `stdout fallback` は便利だが、機密情報を含む例外メッセージを出さないようサニタイズが必要。

### 3) コスト効率・パフォーマンス
- ボトルネックは「225銘柄データ取得」と「LLM入力トークン」。
- 改善優先度:
  1. 非取引日スキップ
  2. 入力特徴量圧縮
  3. キャッシュ（前日データ再利用）
  4. API呼び出し失敗時の段階的縮退

### 4) セキュリティ
- Secret スコープ最小化、依存 pinning、Action SHA pinningは必須。
- Slack出力の mention/URL サニタイズ、ログマスキングを追加推奨。

### 5) より良い代替案
- 代替案A: 「ルールベース選定 + LLMは説明生成のみ」
  - 判定は Python で確定し、Claude は短い根拠文生成のみ担当。
  - コスト・再現性・障害耐性が改善。
- 代替案B: 「単一API呼び出し化」
  - Holdings と Discovery を1リクエストで同時解析。
  - レイテンシ・失敗点を削減（ただしプロンプト設計を厳密化）。
- 代替案C: 「Provider 抽象化」
  - yfinance を primary、別データソースを fallback とするアダプタ設計。

## 総評
設計の方向性は良いですが、実運用に入れるには「セキュリティ最小権限」「データ品質可視化」「LLM出力の厳格バリデーション」の3点を先に固めるべきです。
